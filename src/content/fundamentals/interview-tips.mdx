## Intent / definition

Unlike algorithm interviews, LLD interviews rarely have one "correct" answer — you're being evaluated on *process*:
how you clarify requirements, identify responsibilities, choose patterns deliberately (not by name-dropping), and
discuss tradeoffs. This page is a repeatable framework, not another pattern.

## The problem with going in unprepared

<Callout type="pitfall" title="The most common failure mode">
Jumping straight to code ("let me create a `Vehicle` class...") without first clarifying scope is the single most
common way candidates lose points — even when the code they write is technically fine. Interviewers are watching
whether you ask good questions before committing to a design as much as whether the design itself is good.
</Callout>

## A repeatable framework

### 1. Clarify requirements (2–3 minutes)

Ask about scale, must-have vs. nice-to-have features, and constraints. For "design a parking lot," that means:
how many levels/spots, multiple vehicle types, payment handling, is concurrency (multiple entry gates) in scope?
Write down the functional requirements you agree on before writing a single class.

### 2. Identify the core entities and responsibilities

List the nouns (`ParkingSpot`, `Vehicle`, `Ticket`, `PaymentProcessor`) and, for each, name its *one* responsibility
in a sentence — this is <a href={withBase('/solid/srp/')}>SRP</a> applied at design time, before any code exists.
If a sentence needs "and," that entity probably needs to split into two.

### 3. Sketch relationships before methods

Decide is-a vs. has-a for each relationship (see
<a href={withBase('/fundamentals/composition-vs-inheritance/')}>Composition vs. Inheritance</a>) and draw it —
even a rough box-and-arrow sketch shows the interviewer you're thinking about structure before syntax.

### 4. Apply patterns where they solve a real problem you can name

<Callout type="pitfall" title="Don't pattern-drop">
Naming a pattern because you know its name, without a specific problem it solves in *this* design, reads as
memorization rather than understanding. Instead of "I'll use a Factory here," say "vehicle type determines which
concrete spot-matching strategy to use, and that decision will grow over time, so I'll extract it into a
<a href={withBase('/creational/factory-method/')}>Factory Method</a> instead of an if/elif chain."
</Callout>

### 5. Write the code

Keep it compileable-looking and consistent — real method signatures, real types, no hand-waved pseudocode where a
short real implementation would take the same time.

### 6. Proactively discuss tradeoffs and extensions

Before the interviewer asks, mention: "if we needed to support multiple payment providers later, I'd extract this
into a <a href={withBase('/behavioral/strategy/')}>Strategy</a>," or "this uses a
<a href={withBase('/creational/singleton/')}>Singleton</a> for the spot registry, which would need to become
per-location if we ever supported multiple parking garages." This shows you know the limits of your own design.

## A tiny worked example

Even a "just talk me through it" answer benefits from a few real lines of code to ground the discussion. Here's how
step 2–4 above might look for a small piece of a parking-lot design: matching a vehicle to a compatible spot type is
exactly the kind of decision that grows over time and benefits from Factory Method rather than an if/elif chain.

<CodeTabs>

```python
from enum import Enum
from abc import ABC, abstractmethod


class VehicleType(Enum):
    MOTORCYCLE = "motorcycle"
    CAR = "car"
    BUS = "bus"


class SpotMatcher(ABC):
    @abstractmethod
    def matches(self, spot_size: str) -> bool: ...


class CarSpotMatcher(SpotMatcher):
    def matches(self, spot_size: str) -> bool:
        return spot_size in ("compact", "large")


class SpotMatcherFactory:
    """The one place aware of the mapping — grows without touching calling code."""

    _matchers = {VehicleType.CAR: CarSpotMatcher()}

    @classmethod
    def get_matcher(cls, vehicle_type: VehicleType) -> SpotMatcher:
        return cls._matchers[vehicle_type]
```

```java
enum VehicleType { MOTORCYCLE, CAR, BUS }

interface SpotMatcher {
    boolean matches(String spotSize);
}

class CarSpotMatcher implements SpotMatcher {
    public boolean matches(String spotSize) {
        return spotSize.equals("compact") || spotSize.equals("large");
    }
}

class SpotMatcherFactory {
    // The one place aware of the mapping — grows without touching calling code.
    private static final java.util.Map<VehicleType, SpotMatcher> MATCHERS =
        java.util.Map.of(VehicleType.CAR, new CarSpotMatcher());

    static SpotMatcher getMatcher(VehicleType vehicleType) {
        return MATCHERS.get(vehicleType);
    }
}
```

## Checklist for the last 5 minutes

<div class="pros-cons">
<div>
<h4>✅ Do</h4>
<ul>
<li>Name the SOLID principle or pattern you applied and <em>why</em>, in one sentence each</li>
<li>Mention one thing you'd change with more time (a known simplification)</li>
<li>Ask if the interviewer wants you to go deeper on any specific class</li>
</ul>
</div>
<div>
<h4>❌ Don't</h4>
<ul>
<li>Introduce a pattern with no problem it solves in this specific design</li>
<li>Silently skip clarifying requirements to "save time" — it costs more time later when the design doesn't match expectations</li>
<li>Over-engineer a first pass — start simple, then extend, narrating each extension</li>
</ul>
</div>
</div>

## When this framework applies

- Any "design a system" interview question: parking lot, elevator system, library system, ride-booking, vending
  machine, tic-tac-toe, chess.

## When to deviate

- If the interviewer explicitly asks you to skip straight to a specific class or pattern, follow their lead — the
  framework is a default sequence, not a rigid script.

## Interview talking points

- The single highest-leverage habit: narrate your reasoning out loud continuously, not just at the end. Interviewers
  grade the *process* they can observe, not just the artifact you end with.
- Practice explaining the same design at two levels: a 30-second summary and a 5-minute deep dive — you'll often be
  asked for both.
