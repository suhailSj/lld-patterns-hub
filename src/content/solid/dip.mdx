## Intent / definition

**Dependency Inversion Principle (DIP):** high-level modules (business logic) should not depend on low-level modules
(databases, HTTP clients, frameworks) directly — both should depend on **abstractions**. And abstractions should not
depend on details; details should depend on abstractions. In practice: define the interface from the high-level
module's point of view, and have the low-level module implement it.

## Problem statement

An `AuthService` needs to look up users during login. The obvious first cut has `AuthService` import and instantiate
a `MySqlUserRepository` directly.

## Why the naive design fails

<Callout type="pitfall" title="Business logic welded to infrastructure">

```python
class AuthService:
    def __init__(self):
        self.repo = MySqlUserRepository()  # hard-coded, concrete dependency

    def login(self, email, password):
        user = self.repo.find_by_email(email)
        ...
```

`AuthService` — the important, high-value business logic — now depends directly on a MySQL-specific class. Every
test of login logic needs a real (or heavily mocked) MySQL connection. Switching to DynamoDB, or adding a read-through
cache, means editing `AuthService` itself, even though the login *logic* never actually changed.
</Callout>

This design:

- makes unit testing `AuthService` require spinning up (or intricately faking) a real database;
- couples a business rule ("how login works") to an infrastructure choice ("we use MySQL"), so the two can't evolve
  independently;
- inverts the natural ownership: infrastructure code ends up dictating the shape of business logic, instead of the
  other way around.

## Solution overview

Define a `UserRepository` **interface** owned by the auth/business layer, expressing exactly what login needs:
`find_by_email(email)`. `AuthService` depends only on this interface. `MySqlUserRepository` (and, later,
`DynamoUserRepository`, or `InMemoryUserRepository` for tests) implement it. The dependency arrow is now inverted:
the low-level MySQL class depends on (implements) an interface defined by the high-level module, not the reverse.

## Class / object design

```
   AuthService (high-level)
        │ depends on
        ▼
 «interface» UserRepository       ◀── owned by the business layer
   + find_by_email(email)
        ▲
        │ implemented by
 MySqlUserRepository (low-level) / InMemoryUserRepository (for tests)
```

## Step-by-step explanation

1. Find the concrete, low-level class a high-level class instantiates directly (`MySqlUserRepository()` above).
2. Define an interface expressing only what the high-level class needs from it.
3. Make the low-level class implement that interface instead of the high-level class importing it directly.
4. Have something *outside* both classes (a constructor, a DI container, a `main`/composition root) wire the concrete
   implementation into the high-level class. See <a href={withBase('/fundamentals/dependency-injection/')}>Dependency Injection</a> for the mechanics.

## Python example

<CodeTabs>

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass


@dataclass
class User:
    email: str
    password_hash: str


class UserRepository(ABC):
    """Owned by the business layer — expresses exactly what AuthService needs."""

    @abstractmethod
    def find_by_email(self, email: str) -> User | None: ...


class MySqlUserRepository(UserRepository):
    """A low-level detail. Depends on (implements) the abstraction above."""

    def find_by_email(self, email: str) -> User | None:
        print(f"SELECT * FROM users WHERE email = '{email}'")
        return User(email=email, password_hash="hashed:secret")


class InMemoryUserRepository(UserRepository):
    """A test double — trivial because AuthService never knew about MySQL."""

    def __init__(self, users: dict[str, User]):
        self._users = users

    def find_by_email(self, email: str) -> User | None:
        return self._users.get(email)


class AuthService:
    """High-level module: depends only on the UserRepository abstraction."""

    def __init__(self, repo: UserRepository):
        self._repo = repo

    def login(self, email: str, password_hash: str) -> bool:
        user = self._repo.find_by_email(email)
        return user is not None and user.password_hash == password_hash


# --- usage ---
prod_service = AuthService(MySqlUserRepository())

test_service = AuthService(InMemoryUserRepository({
    "ada@example.com": User("ada@example.com", "hashed:secret")
}))
assert test_service.login("ada@example.com", "hashed:secret") is True
```

```java
import java.util.Map;
import java.util.Optional;

record User(String email, String passwordHash) {}

// Owned by the business layer — expresses exactly what AuthService needs.
interface UserRepository {
    Optional<User> findByEmail(String email);
}

// A low-level detail. Depends on (implements) the abstraction above.
class MySqlUserRepository implements UserRepository {
    public Optional<User> findByEmail(String email) {
        System.out.println("SELECT * FROM users WHERE email = '" + email + "'");
        return Optional.of(new User(email, "hashed:secret"));
    }
}

// A test double — trivial because AuthService never knew about MySQL.
class InMemoryUserRepository implements UserRepository {
    private final Map<String, User> users;
    public InMemoryUserRepository(Map<String, User> users) { this.users = users; }

    public Optional<User> findByEmail(String email) {
        return Optional.ofNullable(users.get(email));
    }
}

// High-level module: depends only on the UserRepository abstraction.
class AuthService {
    private final UserRepository repo;
    public AuthService(UserRepository repo) { this.repo = repo; }

    public boolean login(String email, String passwordHash) {
        return repo.findByEmail(email)
            .map(user -> user.passwordHash().equals(passwordHash))
            .orElse(false);
    }
}

// --- usage ---
// AuthService prodService = new AuthService(new MySqlUserRepository());
//
// AuthService testService = new AuthService(
//     new InMemoryUserRepository(Map.of("ada@example.com", new User("ada@example.com", "hashed:secret"))));
// assert testService.login("ada@example.com", "hashed:secret");
```

</CodeTabs>

**Language notes:** the shape is identical in both languages because DIP is a dependency-direction discipline, not a
language feature. Java's constructor-injection style shown here is the most common; Python projects often achieve
the same thing with plain constructor arguments (as above) or a lightweight DI library — neither language requires a
heavyweight framework to apply DIP.

## Real-world example

Almost every popular web framework's ORM layer is built this way: your service code depends on a repository
interface or query builder abstraction, and the actual SQL dialect (PostgreSQL, MySQL, SQLite) is swapped via
configuration, not code changes. This is also exactly how test suites replace real databases with in-memory fakes
without touching business logic.

## Pros

<div class="pros-cons">
<div>
<h4>✅ Pros</h4>
<ul>
<li>Business logic becomes unit-testable without real infrastructure</li>
<li>Infrastructure (database, HTTP client, message queue) can be swapped without touching business rules</li>
<li>Encourages defining interfaces from the consumer's point of view, which tends to keep them small (ties to ISP)</li>
</ul>
</div>
<div>
<h4>❌ Cons / tradeoffs</h4>
<ul>
<li>Adds an interface and a wiring step for what might have been a one-line direct call</li>
<li>Requires a "composition root" somewhere (constructor calls, a DI container, a <code>main</code> function) that assembles concrete implementations — that wiring code has to live somewhere</li>
</ul>
</div>
</div>

## When to use

- A class directly imports/instantiates a concrete infrastructure class (database driver, HTTP client, file system).
- You want to unit test business logic without spinning up real external systems.

## When to avoid

- For a genuinely single-implementation, unlikely-to-change utility (e.g. a math helper), introducing an interface
  purely for DIP's sake adds ceremony without a real benefit.

<Callout type="tip" title="Interview framing">
DIP is often confused with Dependency Injection — DIP is the <em>design principle</em> (depend on abstractions);
Dependency Injection is one common <em>technique</em> for satisfying it (passing the concrete implementation in from
outside, rather than the class constructing it itself).
</Callout>

## Interview talking points

- State the inversion clearly: it's not just "use interfaces," it's that the *interface is owned by the high-level
  module*, and the low-level module conforms to it — not the other way around.
- Name the direct benefit interviewers listen for: testability without real infrastructure.
- Be ready to distinguish DIP from Dependency Injection when asked — see <a href={withBase('/fundamentals/dependency-injection/')}>Dependency Injection</a>.
