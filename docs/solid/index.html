🧭 SOLID Principles

Five principles that keep object-oriented code easy to extend and hard to break, with a bad-vs-improved example for each.

Single Responsibility Principle

A class should have only one reason to change — one job, one owner, one axis of change.

SOLIDcohesionorder processing

Open/Closed Principle

Software entities should be open for extension, but closed for modification.

SOLIDextensibilitycheckout discounts

Liskov Substitution Principle

Subtypes must be substitutable for their base type without breaking the correctness of the program.

SOLIDinheritancedocument editor

Interface Segregation Principle

No client should be forced to depend on methods it does not use — prefer several small, focused interfaces over one large one.

SOLIDinterfacesplugin systems

Dependency Inversion Principle

High-level modules should not depend on low-level modules; both should depend on abstractions.
