Composite

StructuralGoFfile systems

Compose objects into tree structures and let clients treat individual objects and groups of objects uniformly.

Explanation in simple terms

A file system has files and folders. A folder can contain files and other folders. You want get_size() to work the same way whether you call it on a single file or on a folder containing thousands of nested files and folders — without the caller needing to check “is this a file or a folder?” first.

Practical example

💡 The key insight

Both File and Folder implement the same FileSystemNode interface. A Folder’s get_size() just sums its children’s get_size() — and doesn’t care whether each child is itself a File or another Folder. The recursion falls out naturally from treating both cases uniformly.

Class / object design

        «interface»
       FileSystemNode
      + get_size(): int
      + print_tree(indent)
             ▲
   ┌─────────┴─────────┐
  File               Folder
 (leaf)          (composite: holds a list of FileSystemNode — Files AND Folders)

Python code

from abc import ABC, abstractmethod

class FileSystemNode(ABC):
    @abstractmethod
    def get_size(self) -> int: ...

    @abstractmethod
    def print_tree(self, indent: str = "") -> None: ...

class File(FileSystemNode):
    """Leaf: no children."""

    def __init__(self, name: str, size_bytes: int):
        self.name = name
        self.size_bytes = size_bytes

    def get_size(self) -> int:
        return self.size_bytes

    def print_tree(self, indent: str = "") -> None:
        print(f"{indent}{self.name} ({self.size_bytes}B)")

class Folder(FileSystemNode):
    """Composite: holds other FileSystemNodes — Files or other Folders, indistinguishably."""

    def __init__(self, name: str):
        self.name = name
        self.children: list[FileSystemNode] = []

    def add(self, node: FileSystemNode) -> "Folder":
        self.children.append(node)
        return self

    def get_size(self) -> int:
        # Doesn't care whether each child is a File or a Folder — same call either way.
        return sum(child.get_size() for child in self.children)

    def print_tree(self, indent: str = "") -> None:
        print(f"{indent}{self.name}/")
        for child in self.children:
            child.print_tree(indent + "  ")

# --- usage ---
root = Folder("project")
root.add(File("README.md", 1200))
src = Folder("src")
src.add(File("main.py", 3400)).add(File("utils.py", 900))
root.add(src)

root.print_tree()
print(f"Total size: {root.get_size()} bytes")  # sums recursively through the whole tree

import java.util.ArrayList;
import java.util.List;

interface FileSystemNode {
    long getSize();
    void printTree(String indent);
}

// Leaf: no children.
class FileNode implements FileSystemNode {
    private final String name;
    private final long sizeBytes;
    FileNode(String name, long sizeBytes) { this.name = name; this.sizeBytes = sizeBytes; }

    public long getSize() { return sizeBytes; }
    public void printTree(String indent) {
        System.out.println(indent + name + " (" + sizeBytes + "B)");
    }
}

// Composite: holds other FileSystemNodes — Files or other Folders, indistinguishably.
class Folder implements FileSystemNode {
    private final String name;
    private final List<FileSystemNode> children = new ArrayList<>();
    Folder(String name) { this.name = name; }

    Folder add(FileSystemNode node) { children.add(node); return this; }

    public long getSize() {
        // Doesn't care whether each child is a FileNode or a Folder — same call either way.
        return children.stream().mapToLong(FileSystemNode::getSize).sum();
    }

    public void printTree(String indent) {
        System.out.println(indent + name + "/");
        for (FileSystemNode child : children) {
            child.printTree(indent + "  ");
        }
    }
}

// --- usage ---
// Folder root = new Folder("project");
// root.add(new FileNode("README.md", 1200));
// Folder src = new Folder("src");
// src.add(new FileNode("main.py", 3400)).add(new FileNode("utils.py", 900));
// root.add(src);
//
// root.printTree("");
// System.out.println("Total size: " + root.getSize() + " bytes");

Language notes: both languages express the recursive part-whole relationship identically through a shared interface. Python’s dynamic typing means a Folder.children list can technically hold anything, but the FileSystemNode ABC keeps the contract explicit exactly as Java’s interface does.

Design reasoning

The uniform interface (FileSystemNode) is what makes recursion “free” — Folder.get_size() never needs a type check because every child, leaf or composite, answers get_size() the same way. This is the same idea behind recursive UI component trees (a Panel containing Buttons and other Panels) and organizational charts (a Manager “containing” other employees for reporting/rollup purposes).

Pros

✅ Pros

  • Client code treats individual objects and whole subtrees identically — no type-checking branches
  • Adding new leaf or composite types doesn't require changing existing tree-walking code
  • Naturally recursive operations (size, rendering, search) fall out of the uniform interface

❌ Cons / tradeoffs

  • Can make it harder to restrict what kinds of children a composite accepts (a `Folder` can technically hold anything implementing `FileSystemNode`)
  • Overly general if the hierarchy is shallow and fixed (e.g. exactly one level, never nested) — a plain list may be simpler

When to use

  • Data is naturally tree-shaped (files/folders, UI widgets, menus, organization charts, comment threads) and you want uniform operations across leaves and composites.

When to avoid

  • If the hierarchy never actually nests (it’s really just “a group of items,” not a tree), a plain collection is simpler and avoids the ceremony of a composite/leaf class split.

Interview talking points

  • Identify Composite by the phrase “treat a single object and a group of objects the same way.”
  • Mention that Composite pairs naturally with Iterator (to walk the tree) and Visitor-like traversal for operations that vary by node type without changing the node classes.

Related patterns

  • Decorator — both use recursive, uniform-interface structures, but Decorator models layered behavior around one object, not a part-whole tree.
  • Iterator — commonly used to traverse a Composite tree without exposing its internal structure.
  • Composition vs. Inheritance — Composite is one of the strongest real-world arguments for “has-a” over “is-a” trees.
