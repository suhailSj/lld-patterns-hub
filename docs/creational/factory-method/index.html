Factory Method

CreationalGoFnotification systems

Define an interface for creating an object, but let subclasses (or a parameter) decide which concrete class to instantiate.

What problem it solves

A notification system needs to send messages via email, SMS, or push notification depending on user preference. Client code (the part of the app deciding “send a notification”) shouldn’t need to know the construction details of each notifier — new-ing up an SmtpClient here and a TwilioClient there — and shouldn’t need to change every time a new channel is added.

When to use

  • The exact class to instantiate depends on configuration, user input, or environment, and that decision would otherwise scatter if/elif blocks throughout client code.
  • You want to centralize and name the creation logic so it’s easy to find and extend.
  • You’re following the Open/Closed Principle — adding a new notifier type should mean adding a class, not editing existing client code.

When not to use

  • If there’s only ever going to be one concrete type, a factory is unnecessary ceremony — just construct it directly.
  • Don’t reach for Factory Method to avoid writing a two-line constructor call; use it when the selection logic itself has real complexity or is likely to grow.

Class / object design

       «interface»
        Notifier
      + send(msg)
            ▲
   ┌────────┼─────────┐
EmailNotifier  SmsNotifier  PushNotifier

NotifierFactory
  + create(channel: str) -> Notifier   ◀── the one place that knows every concrete type

Python example

from abc import ABC, abstractmethod

class Notifier(ABC):
    @abstractmethod
    def send(self, message: str) -> None: ...

class EmailNotifier(Notifier):
    def send(self, message: str) -> None:
        print(f"[email] {message}")

class SmsNotifier(Notifier):
    def send(self, message: str) -> None:
        print(f"[sms] {message}")

class PushNotifier(Notifier):
    def send(self, message: str) -> None:
        print(f"[push] {message}")

class NotifierFactory:
    """The one place that knows how to map a channel name to a concrete Notifier."""

    _registry: dict[str, type[Notifier]] = {
        "email": EmailNotifier,
        "sms": SmsNotifier,
        "push": PushNotifier,
    }

    @classmethod
    def create(cls, channel: str) -> Notifier:
        try:
            return cls._registry[channel]()
        except KeyError:
            raise ValueError(f"Unknown notification channel: {channel}")

# --- usage ---
def notify_user(channel: str, message: str) -> None:
    notifier = NotifierFactory.create(channel)  # client code never sees concrete classes
    notifier.send(message)

notify_user("sms", "Your order has shipped!")

interface Notifier {
    void send(String message);
}

class EmailNotifier implements Notifier {
    public void send(String message) { System.out.println("[email] " + message); }
}

class SmsNotifier implements Notifier {
    public void send(String message) { System.out.println("[sms] " + message); }
}

class PushNotifier implements Notifier {
    public void send(String message) { System.out.println("[push] " + message); }
}

// The one place that knows how to map a channel name to a concrete Notifier.
class NotifierFactory {
    static Notifier create(String channel) {
        return switch (channel) {
            case "email" -> new EmailNotifier();
            case "sms" -> new SmsNotifier();
            case "push" -> new PushNotifier();
            default -> throw new IllegalArgumentException("Unknown notification channel: " + channel);
        };
    }
}

class NotificationClient {
    // client code never sees concrete classes
    static void notifyUser(String channel, String message) {
        Notifier notifier = NotifierFactory.create(channel);
        notifier.send(message);
    }
}

// --- usage ---
// NotificationClient.notifyUser("sms", "Your order has shipped!");

Language notes: the Python version uses a dictionary registry, which makes registering a new channel a one-line addition and is easy to extend at runtime (e.g. plugin systems registering their own notifier at import time). Java’s switch expression (14+) is concise for a fixed, closed set of types; for an open-ended/plugin scenario, Java typically uses a Map<String, Supplier<Notifier>> registry instead, mirroring the Python dictionary approach.

Real-world example

java.util.Calendar.getInstance() is a Factory Method — it returns a GregorianCalendar (or a locale-specific subclass) without callers ever naming the concrete class. Similarly, Python’s logging.getLogger(name) hands back a concrete Logger instance through a factory function that hides caching/construction details.

Advantages and tradeoffs

✅ Advantages

  • Client code depends only on the `Notifier` interface, not any concrete class
  • Adding a new channel means adding one class + one registry entry — no existing code changes
  • Centralizes and names the "which concrete type do I need" decision instead of scattering it

❌ Tradeoffs

  • Adds a layer of indirection for what might be a very simple constructor call
  • A registry/switch still needs *some* central place aware of every concrete type — it doesn't eliminate that knowledge, just concentrates it in one spot
💡 Interview framing

Factory Method is often the very first pattern interviewers expect you to reach for when they say “how would you avoid a giant if/else picking which class to construct?” Name it, and mention the registry/dictionary variant as the more extensible real-world version of the textbook subclassing version.

Related patterns

  • Abstract Factory — a step up: creates families of related objects instead of one product.
  • Strategy — often paired with Factory Method: the factory decides which strategy to hand back.
  • Open/Closed Principle and Dependency Inversion Principle — Factory Method is one of the main tools for satisfying both.
