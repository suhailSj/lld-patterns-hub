Composition vs. Inheritance

FundamentalsOOPdelivery vehicles

Two ways to reuse code and share behavior — 'is-a' inheritance vs. 'has-a' composition — and why composition is usually the safer default.

Intent / definition

Two classes can share behavior in two fundamentally different ways. Inheritance (“is-a”): a subclass extends a base class and inherits its implementation directly — ElectricBike extends Vehicle. Composition (“has-a”): a class holds a reference to another object and delegates to it — DeliveryVehicle has a MovementBehavior. Both achieve reuse; they differ in how tightly coupled the reusing class becomes to the reused implementation.

Problem statement

A delivery company has vehicles: bikes, vans, and drones. Each moves differently (pedal power, driving, flying) and some can carry different amounts of cargo. New vehicle types are added a few times a year.

Why the naive (inheritance-heavy) design fails

🚫 Deep hierarchies get brittle fast

Modeling this with inheritance — Vehicle → LandVehicle → Bike → ElectricBike — works until a requirement doesn’t fit the tree. What about a vehicle that can both drive and fly (a delivery drone that also has wheels for the last few feet)? Multiple inheritance of behavior gets messy fast, and every vehicle is locked into whatever movement logic its ancestors chose — you can’t swap ElectricBike’s movement behavior at runtime without changing its class.

This design:

  • creates a “fragile base class” problem — a change to Vehicle can ripple unpredictably through every descendant;
  • forces movement and cargo logic to be decided at compile time (which class you picked), not configuration time;
  • struggles with any vehicle that needs a combination of behaviors that don’t map cleanly onto a single-inheritance tree.

Solution overview

Model “how it moves” and “how much it can carry” as separate, swappable objects (MovementBehavior, CargoBehavior) that a DeliveryVehicle holds, rather than inherits. Each vehicle type is composed from whichever behaviors it needs — including combinations that wouldn’t fit cleanly in a single-inheritance tree.

Class / object design

DeliveryVehicle
  - movement: MovementBehavior     ◀── composition ("has-a")
  - cargo: CargoBehavior           ◀── composition ("has-a")
  + deliver(destination)

«interface» MovementBehavior       «interface» CargoBehavior
  + move(to)                          + load(item)
       ▲                                   ▲
  PedalMovement / DriveMovement / FlyMovement    SmallCargo / LargeCargo

Step-by-step explanation

  1. Identify the behaviors that vary independently of “what kind of vehicle this is” (movement, cargo capacity).
  2. Extract each as a small interface (MovementBehavior, CargoBehavior).
  3. Give DeliveryVehicle references to instances of these interfaces instead of inheriting from a movement-specific base class.
  4. Assemble each concrete vehicle by choosing which behaviors to plug in — mixing and matching freely, including combinations a class hierarchy couldn’t express.

Python example

from abc import ABC, abstractmethod

class MovementBehavior(ABC):
    @abstractmethod
    def move(self, destination: str) -> str: ...

class PedalMovement(MovementBehavior):
    def move(self, destination: str) -> str:
        return f"Pedaling to {destination}"

class DriveMovement(MovementBehavior):
    def move(self, destination: str) -> str:
        return f"Driving to {destination}"

class FlyMovement(MovementBehavior):
    def move(self, destination: str) -> str:
        return f"Flying to {destination}"

class DeliveryVehicle:
    """Composed FROM behaviors (has-a), not built as a deep inheritance tree (is-a)."""

    def __init__(self, name: str, movement: MovementBehavior):
        self.name = name
        self._movement = movement

    def deliver(self, destination: str) -> None:
        print(f"{self.name}: {self._movement.move(destination)}")

    def set_movement(self, movement: MovementBehavior) -> None:
        # A behavior can be swapped at runtime — impossible with a fixed inheritance choice.
        self._movement = movement

# --- usage ---
bike = DeliveryVehicle("Cargo Bike #4", PedalMovement())
bike.deliver("123 Main St")

drone = DeliveryVehicle("Drone #9", FlyMovement())
drone.deliver("456 Oak Ave")

# A bike upgraded with an electric assist just swaps its movement behavior:
bike.set_movement(DriveMovement())

interface MovementBehavior {
    String move(String destination);
}

class PedalMovement implements MovementBehavior {
    public String move(String destination) { return "Pedaling to " + destination; }
}

class DriveMovement implements MovementBehavior {
    public String move(String destination) { return "Driving to " + destination; }
}

class FlyMovement implements MovementBehavior {
    public String move(String destination) { return "Flying to " + destination; }
}

// Composed FROM behaviors (has-a), not built as a deep inheritance tree (is-a).
class DeliveryVehicle {
    private final String name;
    private MovementBehavior movement;

    DeliveryVehicle(String name, MovementBehavior movement) {
        this.name = name;
        this.movement = movement;
    }

    void deliver(String destination) {
        System.out.println(name + ": " + movement.move(destination));
    }

    void setMovement(MovementBehavior movement) {
        // A behavior can be swapped at runtime — impossible with a fixed inheritance choice.
        this.movement = movement;
    }
}

// --- usage ---
// DeliveryVehicle bike = new DeliveryVehicle("Cargo Bike #4", new PedalMovement());
// bike.deliver("123 Main St");
//
// DeliveryVehicle drone = new DeliveryVehicle("Drone #9", new FlyMovement());
// drone.deliver("456 Oak Ave");
//
// bike.setMovement(new DriveMovement()); // upgraded with an electric assist

Language notes: this “favor composition” guidance applies identically in both languages — it’s a design discipline, not a language feature. Java developers sometimes reach for inheritance more readily because abstract classes/interfaces feel like “the OOP way”; Python’s flexibility (duck typing, first-class functions) often makes composition even more natural, since a MovementBehavior could just be a function passed in rather than a full class.

Real-world example

Game engines almost universally use composition (“entity-component systems”) instead of deep inheritance for exactly this reason: a game object’s Health, Movement, Renderer, and Inventory are separate, swappable components, because a FlyingArmoredTank and a WalkingUnarmoredScout don’t fit cleanly into one inheritance tree, but freely compose from shared components.

Pros of composition

✅ Composition pros

  • Behavior can be swapped at runtime, not fixed at class-definition time
  • Avoids deep, fragile inheritance chains
  • Naturally supports combinations that wouldn't fit a single-inheritance tree

❌ Composition cons

  • More objects and wiring to keep track of
  • Slightly more indirection to read ("what does this vehicle actually do?" requires checking which behavior objects it holds)

When to use inheritance instead

Inheritance is still the right tool when the relationship is genuinely, permanently an “is-a” relationship that satisfies Liskov Substitution — e.g. SavingsAccount extends BankAccount when every operation valid on BankAccount is also valid, unmodified, on SavingsAccount.

When to avoid composition

  • Extremely simple, unlikely-to-vary behavior doesn’t need to be extracted into a swappable component — that’s premature abstraction.
💡 Interview framing

The famous guidance is “favor composition over inheritance,” not “never use inheritance.” Be ready to say when inheritance is still appropriate — a genuine, stable is-a relationship — rather than treating it as universally wrong.

Interview talking points

  • State the rule of thumb precisely: prefer composition by default; use inheritance only for a true is-a relationship that will hold up under Liskov Substitution.
  • Nearly every structural/behavioral pattern built on delegation (Decorator, Bridge, Strategy) is, at its core, “apply composition instead of inheritance to solve a specific flexibility problem” — naming that connection shows deeper understanding.

Related patterns

  • Strategy and Decorator — both are composition applied to specific problems (swappable algorithms, stackable behavior).
  • Bridge — composition used specifically to decouple two independently varying dimensions.
  • Template Method — the main pattern that deliberately leans on inheritance instead; useful to contrast.
